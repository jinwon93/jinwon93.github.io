예약 앱 QA에서 "화면이 너무 확확 바뀌어서 보기 불편하다"는 체감 피드백이
들어왔다. 감각적인 표현이지만 원인은 구조적이었다 — 달력 날짜를 누를 때마다
**대형 상태 리스트가 통째로 교체**되고 있었다.

## 원인: 리스트 교체 = 전면 recompose

예약 목록을 `mutableStateOf`로 들고 있는데, 날짜 선택 시 서버에서 받은 새
리스트로 **참조 자체를 교체**했다. Compose 입장에선 상태가 바뀌었으니 이를
읽는 컴포저블 전부에 recompose가 전파된다. 예약이 많은 날짜일수록 리스트가
크고, 교체 빈도가 잦을수록 화면이 출렁인다.

더 나쁜 건 입력 쪽이었다. 달력에서 날짜를 연타하거나 드래그하듯 이동하면
`updateSelectedDate()`가 그때마다 호출돼 **같은 날짜에 대한 API 호출이
중복 발사**됐다. 네트워크 응답이 뒤섞여 도착하면 마지막에 도착한 것이
이긴다 — 선택한 날짜와 다른 데이터가 그려질 수도 있는 구조다.

## 해결: 입력을 정돈하고 나서 상태를 바꾼다

```kotlin
// 날짜 선택 스트림 정돈
selectedDateFlow
    .debounce(300)              // 연타 흡수 — 마지막 선택만 유효
    .distinctUntilChanged()     // 같은 날짜 재선택 시 API 재호출 차단
    .collectLatest { loadReservations(it) }
```

- **debounce 300ms**: 빠르게 훑는 동안엔 로드하지 않고, 멈춘 날짜만 로드
- **distinctUntilChanged**: 동일 날짜 중복 호출 제거
- `collectLatest`: 이전 로드가 진행 중이면 취소 — 늦게 도착한 응답이
  화면을 덮어쓰는 역전 문제도 함께 사라진다

상태 쪽도 통째 교체 대신 변경분 위주로 다루도록 정리해 recompose 범위를 줄였다.

## 검증은 체감으로도, 계측으로도

수정 후 같은 시나리오(날짜 연타, 예약 많은 날짜 이동)에서 API 호출 수가
입력 횟수가 아니라 **멈춘 횟수**만큼만 나가는 것을 확인했다. "확확 바뀐다"던
체감 피드백도 사라졌다. 성능 이슈는 프로파일러 수치로 잡지만, 시작은 대개
이런 비정형 체감 제보다 — 무시하지 말 것.

## 배운 것

- Compose 성능 문제의 전형은 **큰 상태의 잦은 통째 교체**다. 상태의 입도(粒度)를
  쪼개고, 교체 빈도를 입력단에서 제어하라.
- debounce·distinctUntilChanged·collectLatest 3종은 "사용자 입력 → 네트워크 →
  상태" 파이프라인의 기본 위생이다.
- 응답 역전(늦은 응답이 화면을 덮는 것)은 취소로 푼다. 순서 보장을 바라지 말 것.
