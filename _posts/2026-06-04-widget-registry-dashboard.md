통계 대시보드에 "계정별 커스터마이징"을 넣었다 — 탭 순서·표시 여부, 위젯
순서·숨김·차트 종류·가로폭을 사용자마다 저장한다. 이런 기능은 구현보다
**구조**가 수명을 결정한다. 화면을 하드코딩한 채 커스터마이징을 얹으면
탭 하나 추가할 때마다 편집 UI·저장 로직·렌더러를 전부 고치게 된다.

## 핵심: 화면 명세를 타입 레지스트리로

페이지별 위젯 정의를 순수 데이터로 선언했다. 위젯의 종류(KPI/차트),
지원 차트 타입 목록, 허용 가로폭 — 이 레지스트리가 렌더러·편집 UI·API
검증이 공유하는 **단일 소스**다.

```ts
{ id: "chart_revenue_trend", kind: "chart",
  charts: ["line", "area", "bar", "composed"], spans: [10, 5] }
```

사용자 저장값(jsonb)은 신뢰하지 않는다. 로드 시 `mergeLayout`이 레지스트리와
대조해 **정규화**한다 — 삭제된 위젯 제거, 신규 위젯 자동 추가, 허용값 밖
차트타입·폭은 기본값으로 보정. 덕분에 스키마가 진화해도 기존 사용자 설정이
깨지지 않고, 신규 위젯 추가는 레지스트리에 객체 하나 넣는 것으로 끝난다.

## 설계 디테일 셋

**① locked 탭.** 핵심 탭 5개는 숨길 수 없고 순서만 바꿀 수 있다. 자유도를
100% 주면 "대시보드가 텅 빈" 지원 문의가 온다. 자유와 가드레일의 균형.

**② 서버 주입으로 플리커 제거.** 탭 설정을 클라이언트에서 fetch하면 기본
구성이 먼저 그려졌다가 재배열되는 깜빡임이 생긴다. SSR에서 설정을 읽어
첫 페인트부터 사용자 구성으로 렌더 — 커스터마이징 기능의 완성도는 이런 데서 갈린다.

**③ 편집 범위의 계층화.** 풀편집(순서·표시·차트·폭) 7개 탭, 차트 종류만
바꾸는 인라인 편집 3개 탭, 대상 외(텍스트 리포트·실시간) 2개 탭 — 전부에
같은 자유도를 주는 대신 탭 성격에 맞는 수준만 열었다.

## 차트 13종과 의존성 0의 지도

차트 렌더러는 13종(line/bar/donut/radar/treemap/composed...)을 지원하되,
**위젯 데이터 특성에 맞는 타입만** 선택지로 노출한다 — 시계열엔 line/area/bar,
구성비엔 donut/treemap. "모든 위젯에 모든 차트"는 이상한 조합(히트맵을
파이로?)을 사용자에게 떠넘기는 것이다.

지역 지도 위젯은 라이브러리를 넣는 대신 한국 시도 좌표 기반의 경량 SVG 버블
지도로 직접 그렸다 — 외부 의존성 0. 필요한 게 "시도별 버블 하나"라면
지도 라이브러리 수백 KB는 과하다.

## 배운 것

- 커스터마이징 기능의 본질은 UI가 아니라 **저장값과 코드 정의의 정합화
  (merge) 전략**이다. 이게 없으면 스키마 변경이 곧 사용자 데이터 파손이다.
- 화면 명세를 데이터로 내리면 "기능 추가 = 등록 1건"이 된다. 반대로 하면
  "기능 추가 = 전면 수정"이다.
- 자유도는 계층화해서 줘라. locked 탭 같은 가드레일이 없는 커스터마이징은
  지원 비용으로 돌아온다.
