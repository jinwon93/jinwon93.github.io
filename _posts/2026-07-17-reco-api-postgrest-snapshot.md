테이블오더 앱에 넣을 매장별 메뉴 추천 API를 만들었다. 처음엔 Edge Function으로
시작했다가 중간에 방향을 틀었다 — **함수 호출당 과금이 붙고, 앱 개발자에게 넘길
인터페이스로도 과했다.** 결론은 DB가 이미 제공하는 REST(PostgREST RPC)로 회귀.
이 글은 그 설계 전환과, 도중에 만난 297만 행 셀프조인 이야기다.

## 설계: 조회 시점에 계산하지 않는다

추천은 4종 — 인기 / 급상승 / 시간대 맞춤 / 함께 주문. 전부 **선계산 스냅샷** 구조다.

- 야간 배치가 스냅샷 테이블 2개를 빌드 (인기·급상승·시간대 / 함께주문 연관규칙, lift 기반)
- 조회는 단일 RPC `get_store_recommendations(매장, 기준메뉴?)` — 요일·시간대
  (아침/점심/저녁/심야)는 서버가 KST 기준 자동 판정
- 함수를 STABLE로 선언해 **GET + 쿼리 파라미터** 호출 가능 → CDN 캐시를 공짜로 얻음
- 앱 입장에선 anon 키 헤더 하나로 끝. 품절·미사용 메뉴는 마스터 조인으로 자동 제외
- 갱신은 pg_cron 새벽 배치. "시간대 맞춤이 최대 하루 늦을 수 있다"는 트레이드오프는
  문서에 명시하고 수용

응답 2~4KB 기준으로 계산해보니 무료 egress 한도 안에서 월 수천만 호출이 가능했다.
호출량 기반 과금 요소가 0인 추천 API가 됐다.

## 고비: 30일 297만 행 셀프조인 180초

"함께 주문" 연관규칙은 같은 주문에 담긴 메뉴 페어를 세야 한다. 순진한 셀프조인은
30일치 297만 행에서 180초를 넘겨 타임아웃됐다. 세 가지로 잡았다.

1. **임시테이블 3단**: 유효 주문 → 주문·메뉴 정제본 → 페어 집계로 단계를 나누고
   각 단계에 복합 인덱스 + `ANALYZE`
2. **비대칭 페어**: `s1 < s2` 조건으로 (A,B)/(B,A) 중복을 제거 — 조인량이 절반이 된다
3. 페어의 가산성(일별로 더해도 총합 동일)을 이용해 이후 증분 빌드로 전환

최종 스냅샷 90,807행. 검증 매장에서 인기 8·시간대 5·함께주문 5건, lift 값 정상 확인.

## 배운 것

- **서버리스 함수가 기본값이 아니다.** DB가 REST를 이미 제공하면(PostgREST)
  계층 하나를 통째로 없앨 수 있다 — 과금도, 배포물도, 장애 지점도 함께 사라진다.
- 추천처럼 "약간 낡아도 되는" 데이터는 조회 시 계산 금지. 선계산 + 캐시 가능한
  GET이 정석이다.
- 대량 셀프조인은 (정제 단계 분리, 비대칭 조건, 가산성 증분) 세 무기로 잡는다.
- 트레이드오프(하루 지연)는 숨기지 말고 API 명세에 적어라. 소비자가 판단하게.
