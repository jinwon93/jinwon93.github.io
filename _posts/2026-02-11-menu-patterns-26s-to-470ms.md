"이 메뉴는 어떤 요일, 어떤 시간대에 잘 팔리나"를 보여주는 메뉴 패턴 API가
**26초**씩 걸렸다. 타임아웃 직전의 숫자다. 원인과 해법 모두 교과서적이라 기록해둔다.

## 원인: 호출마다 306만 행 조인

실행 계획을 보니 주문 상세(306만 행, 1.15GB)와 주문(124만 행)을 **호출할 때마다**
전체 매장 범위로 조인·풀스캔하고, 요일·시간대 추출과 그룹핑을 즉석에서 하고
있었다. `temp written` — 정렬이 메모리를 넘겨 디스크까지 흘렀다.

요일과 시간대는 주문 시각에서 파생되는 값이다. 조회 시점에 매번 계산할 이유가
전혀 없다.

## 해법: 그레인을 정하고 미리 굽는다

`(매장, 날짜, 요일, 시간대, 메뉴)` 그레인의 사전집계 테이블을 신설했다.

- 요일(dow)·시간대(hour_band: 아침/점심/저녁/심야, KST 기준 4구간)를
  **집계 시점에 미리 계산**해 컬럼으로 저장
- 메뉴명 정규화 컬럼은 STORED generated column — 입력 시 자동 계산
- 빌더 함수는 일 단위 `DELETE + INSERT` **멱등** 처리 (같은 날짜를 몇 번 돌려도 동일 결과),
  `statement_timeout`·`work_mem` 명시로 폭주 방지
- 조회 RPC는 집계 테이블만 읽도록 재작성, 별칭 병합 JOIN 통합,
  `MODE() WITHIN GROUP` → `MAX()`로 그룹당 정렬 제거
- pg_cron 새벽 배치 등록 — 일별 리포트 컴파일 직후 슬롯

## 검증이 배포 조건

32일치 백필(37.7만 행, 94MB) 후 **raw 대비 합계 검증**을 돌렸다 —
수량 4,477,920 / 매출 490.5억이 소수점까지 일치. 집계 테이블이 원본과 같음을
증명한 뒤에야 API를 전환했다.

| 지표 | 전 | 후 |
|---|---|---|
| 응답 시간 | 26s | **470ms (55배)** |
| 스캔 대상 | 306만 행 조인 + 디스크 정렬 | 집계 테이블 직독 |
| 파생값 계산 | 호출마다 | 집계 시 1회 |

## 배운 것

- 파생값(요일·시간대)을 조회 시점에 계산하고 있다면 그게 신호다 —
  **그레인을 정의하고 미리 구워라.**
- 사전집계 테이블의 생명은 멱등 빌더다. 재실행이 두려운 배치는 운영할 수 없다.
- "빨라졌다"는 반쪽 증명이다. **원본과 합계가 같다**까지 확인하고 전환하라.
