매장 운영 시스템의 주문 데이터로 통계 대시보드를 만들게 됐다. 가장 먼저 정한
원칙은 하나 — **통계 쿼리가 운영 DB에 단 한 번도 직접 붙지 않는다.** 매장의
주문 처리와 사장님의 통계 조회가 같은 DB에서 경합하는 순간 둘 다 불행해진다.

## 구조: 복제 → 집계 → 리포트

분석용 DB(Supabase)를 따로 두고 3단 계층으로 설계했다.

```
운영 DB ──sync──▶ raw_*  ──집계──▶ analytics_* ──컴파일──▶ report
              (원본 복제)      (일별/매장별/메뉴별)     (대시보드 최종)
```

- **raw**: 운영 DB 테이블의 복제본. UPSERT만 허용, 직접 수정 금지 (재동기화로만 고침)
- **analytics**: 일별×매장별×메뉴별 그레인으로 미리 집계
- **report**: 대시보드가 바로 읽는 최종 산출물

대시보드는 report/analytics만 읽는다. raw를 직접 SELECT하는 쿼리는 만들지 않는다 —
이 규칙 하나가 이후 모든 성능 문제의 방어선이 된다.

## 동기화에서 배운 것들

마스터 데이터(매장·메뉴)부터 옮기는데 메뉴 아이템이 46,783건이었다.
API 게이트웨이의 응답 행 수 제한에 걸려서 1,000건 단위 페이징 헬퍼(`fetchAll`)를
만들어 전량을 처리했다. **"전부 가져왔다"고 믿지 말고 행 수를 세어 검증하라** —
limit에 잘려도 에러는 나지 않는다.

운영 원칙 몇 가지를 처음부터 박았다.

- 모든 동기화는 `sync_logs`에 기록 — 언제, 몇 건, 성공/실패
- 동기화 API는 Authorization 헤더로 보호 (cron 전용 시크릿)
- cron 체인은 fault-tolerant — 한 단계 실패가 전체를 멈추지 않게
- 매일 새벽 Vercel Cron이 어제 데이터를 동기화 → 집계 → 리포트 컴파일 순서로 체인 실행

## 배운 것

- OLTP/OLAP 분리는 규모가 커진 뒤가 아니라 **첫 삽부터** 하는 게 싸다.
  나중에 분리하려면 이미 raw를 직접 읽는 쿼리 수십 개가 발목을 잡는다.
- raw 계층의 "UPSERT만, 수정 금지" 규칙은 데이터 사고 시 복구를 단순하게 만든다 —
  언제든 재동기화하면 원본과 같아진다.
- 페이징·로그·인증 같은 지루한 기반이 파이프라인의 수명을 결정한다.
