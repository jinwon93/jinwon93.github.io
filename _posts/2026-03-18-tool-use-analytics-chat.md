"강남 매장들 주류 비중 어때?"라고 물으면 답이 나오는 데이터 분석 챗을 만들었다.
구조는 단순하다 — LLM에게 집계 RPC 5종을 도구로 쥐여주고, 질문을 도구 호출로
번역하게 한다. 단순한 구조를 실서비스로 만드는 건 세 가지 설계 결정이었다.

## 결정 1: 권한은 LLM이 아니라 디스패처가 지킨다

제일 중요한 원칙. **LLM에게 "이 사용자는 A매장만 볼 수 있어"라고 알려주고
알아서 지키길 기대하면 안 된다.** 프롬프트는 보안 경계가 아니다.

- 요청 진입 시 서버가 사용자의 접근 가능 매장 목록을 확정
- 도구 디스패처가 모든 RPC 호출에 이 목록을 **강제 주입** — LLM이 다른 매장을
  요청해도 교집합 필터로 잘려나간다
- 지역 검색(pg_trgm 기반 자연어 주소 매칭)도 같은 권한 필터를 통과

LLM은 "무엇을 조회할지"만 정하고, "무엇을 조회할 수 있는지"는 코드가 정한다.

## 결정 2: 수동 tool-use 루프 + 이중 가드

프레임워크 의존성을 늘리는 대신 SDK 위에 수동 루프를 짰다. 루프에는 가드 둘:

- **MAX_STEPS 8** — 도구를 8번 부르고도 결론이 없으면 중단. 무한 루프·비용 폭주 방지
- **12KB payload 가드** — 도구 결과가 크면 잘라서 전달. 대량 행을 그대로 넣으면
  토큰 비용이 터지고 답변 품질도 오히려 떨어진다

모든 대화는 감사 로그로 자동 기록된다. 이 로그가 나중에 품질 플래그 파이프라인
(환각 탐지)의 원천이 됐다.

## 결정 3: 프롬프트 캐싱 — 같은 걸 매번 다시 읽지 않는다

시스템 프롬프트와 도구 정의는 매 요청 동일하다. 그런데 기본 상태로는 매 호출마다
이 수천 토큰을 새로 과금당한다. cache_control 마킹으로 시스템 프롬프트와 도구
블록을 캐시화했다 — tool-use 루프의 이어지는 스텝들도 같은 캐시를 공유한다.

효과를 믿음이 아니라 계측으로 확인하기 위해, 응답에 캐시 토큰 통계를 실어
디버그 모드에서 메시지마다 표시하게 했다:

```
in 3245 (cache 87%: r2820/w0) · out 312
```

읽기 토큰의 87%가 캐시 히트 — 반복 질의가 많은 챗 특성상 비용 절감 폭이 크다.

## 배운 것

- LLM 시스템의 권한 경계는 **항상 도구 실행 계층**에 둔다. 프롬프트의 지시는
  UX이지 보안이 아니다.
- 루프 상한과 payload 상한은 처음부터 박아라. "설마"가 청구서로 돌아온다.
- 캐싱 같은 최적화는 **계측 UI를 같이** 만들어야 유지된다. 보이지 않는 절감은
  다음 리팩토링 때 조용히 깨진다.
