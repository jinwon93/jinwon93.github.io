데이터 분석 챗의 기능이 늘수록 응답이 무거워졌다. 도구를 여러 번 부르는
질문은 최종 답까지 20초 — 기능적으론 정답인데 체감으론 낙제였다.
모델을 바꾸지 않고 체감을 고친 작업들을 정리한다.

## 스트리밍: 마지막 스텝만 흘려보낸다

tool-use 루프는 중간 스텝(도구 호출 JSON)과 최종 스텝(사용자에게 보여줄 답변)의
성격이 다르다. 중간 스텝은 **파싱해야 하므로 non-stream 유지**, 최종 답변만
SSE 스트리밍으로 전환했다. 클라이언트는 ReadableStream을 파싱해 글자를
점진 렌더링한다.

총 소요 시간은 같지만 **첫 글자까지의 시간(TTFT)이 20초 → 3초.** 사용자는
"기다리는 20초"가 아니라 "읽는 20초"를 경험한다. 지연 최적화에서 총량보다
분포(첫 응답)가 중요할 때가 많다.

## 답변 캐시: 같은 질문은 다시 계산하지 않는다

"오늘 매출 얼마야?"는 하루에도 여러 사용자가 반복해서 묻는다.
질문을 정규화(공백·대소문자)하고 해시를 키로 **10분 TTL 답변 캐시**를 뒀다 —
중복 질의의 약 20%가 LLM 호출 없이 즉답으로 빠진다.

프롬프트 캐싱(시스템·도구 블록)이 "입력 토큰의 재사용"이라면, 이건
"출력 전체의 재사용"이다. 층이 다르니 둘 다 두는 게 맞다. TTL을 10분으로
짧게 잡은 건 통계 데이터의 신선도 특성 때문 — 캐시는 항상 데이터의 갱신
주기와 짝지어 설계한다.

## 운영: 로그 대시보드와 알림

- 전 대화 로그(질문/답변/도구 호출/소요 시간)를 리뷰하는 관리자 페이지 —
  품질 플래그 파이프라인과 같은 로그를 다른 각도로 보는 화면
- 루프 상한 초과(tool_loop_exceeded) 같은 이상 신호는 Slack 웹훅으로 즉시 알림 —
  기존 DB 알림 함수를 재활용해 배관 비용 0

## 품질: 감이 아니라 회귀 테스트로

프롬프트를 고칠 때마다 "좋아진 것 같은데?"로 배포하는 건 도박이다.

- **few-shot 예시**: 잘 나온 Q&A를 시스템 프롬프트에 박아 표 형식·단위 표기
  같은 답변 패턴을 고정
- **eval 세트 20문항**: 정답이 있는 질문들로 프롬프트 수정 전후를 비교하는
  회귀 테스트. 20개는 작지만, 0개와 20개의 차이는 질적으로 다르다
- **chart_hint**: 도구 결과에 차트 힌트 메타를 실어 프론트가 적절한 차트를
  자동 렌더 — "표로 줄까 차트로 줄까"를 모델의 재량에서 데이터의 속성으로 이동

## 배운 것

- 체감 성능의 단위는 총 소요가 아니라 **TTFT**다. 스트리밍은 기능이 아니라 UX다.
- 캐시는 층별로: 프롬프트 캐시(입력) + 답변 캐시(출력). 키 설계(정규화)와
  TTL(데이터 신선도)이 본체다.
- LLM 프롬프트도 코드다 — 수정에는 회귀 테스트(eval)가 따라붙어야 한다.
