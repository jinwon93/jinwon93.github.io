사용자별 맞춤 대시보드(위젯 순서·차트 종류를 계정마다 저장)를 확장하다 보니
호출되는 집계 쿼리가 늘었고, 무거운 놈 둘이 수면 위로 올라왔다. 각각 다른
교과서적 패턴이라 기록해둔다.

## 1. 1.29GB 테이블, 버퍼 10만 → 2,740

메뉴 일별 집계 테이블은 360만 행, 1.29GB. 랭킹 쿼리가 인덱스를 타고도
**heap 접근** 때문에 버퍼를 10만 페이지씩 읽고 있었다. 인덱스에서 위치를 찾고
실제 값은 테이블 본체에서 가져오는 구조라, 데이터가 흩어져 있으면 randam I/O가 폭발한다.

해법은 쿼리가 필요로 하는 컬럼을 인덱스에 다 넣는 **커버링 인덱스** —
Index Only Scan으로 전환해 heap 접근 자체를 없앴다.

| 지표 | 전 | 후 |
|---|---|---|
| 버퍼 읽기 | 100,600 페이지 | **2,740 (36배↓)** |
| 읽기량 | 794MB | **~22MB** |
| 콜드 응답 | ~10초 | **~1초** |

인덱스 크기 338MB는 대가다. 읽기 794MB→22MB와의 교환이면 남는 장사지만,
쓰기가 많은 테이블이었다면 다른 답을 골랐을 것이다.

## 2. 서브쿼리가 24,214번 돈다

예약 통계 쿼리는 매장별로 상관 서브쿼리를 반복 실행하고 있었다.
실행 계획의 loops 수치가 **24,214** — 매장 수만큼 같은 집계를 다시 하는 구조다.

```sql
-- before: 매장마다 서브쿼리 재실행 (loops=24,214)
SELECT s.sto_no,
  (SELECT count(*) FROM bookings b WHERE b.sto_no = s.sto_no AND ...) AS cnt
FROM stores s;

-- after: 한 번 집계해서 조인 (파생 테이블)
SELECT s.sto_no, COALESCE(b.cnt, 0)
FROM stores s
LEFT JOIN (SELECT sto_no, count(*) AS cnt FROM bookings WHERE ... GROUP BY sto_no) b
  USING (sto_no);
```

**3,586ms → 47ms (76배).** 코드 변경은 사실상 조인 위치 이동뿐이다.

## 클라이언트 쪽도 한 줄씩

서버만 잡을 게 아니라 부르는 쪽도 정리했다 — React Query `staleTime` 30초→5분,
폴링 분당 8회→1.5회. 대시보드류 데이터는 분 단위 신선도면 충분한데
초 단위로 다시 묻고 있었다.

## 배운 것

- `EXPLAIN (ANALYZE, BUFFERS)`에서 **버퍼 수치와 loops**를 먼저 보라.
  느린 쿼리의 8할은 이 둘 중 하나다.
- 커버링 인덱스는 공간과 읽기의 트레이드오프다. 수치로 저울질하고 결정하라.
- 상관 서브쿼리 → 파생 테이블 조인은 가장 싸고 극적인 리팩토링이다.
- 캐시 정책(staleTime)은 성능 튜닝의 일부다. 데이터의 실제 갱신 주기에 맞춰라.
