데이터 분석 AI 챗(자연어 질문 → 집계 RPC 호출 → 수치 근거 답변)을 운영하면서
가장 무서운 실패 유형은 오류가 아니라 **"검색한 척"**이다. 도구를 호출하지도 않고
"해당 매장은 없습니다"라고 단정하는 답 — 사용자는 검증할 방법이 없다.
이걸 사람이 아니라 SQL이 잡게 만든 파이프라인을 정리한다.

## 설계: 모든 대화를 구조화 로그로

챗의 모든 턴을 로그 테이블에 남긴다 — 질문, 응답, **호출한 도구 목록과 결과 행 수**,
소요 시간. 그 위에 품질 플래그 뷰를 정의했다. 핵심 플래그 3종:

- **no_tool_negative**: 도구 호출 0회인데 응답이 "없다/불가능" 계열 단정 → 환각 최우선 의심
- **empty_result**: 도구는 호출했지만 결과 0건 → 데이터 커버리지(태깅 누락 등) 개선 신호
- **tool_failed**: 도구 실행 오류

포인트는 분류다. 셋 다 "이상한 답"이지만 처방이 전혀 다르다 — 첫째는 프롬프트 규칙,
둘째는 데이터 보강, 셋째는 코드 수정.

## 운영: 주간 자동 리포트 폐루프

매주 자동 작업이 플래그를 집계해 전주와 비교 리포트를 만든다. 실제 한 주의 결과:

- 대화 5건 중 플래그 3건 (60%) — 전주 30%에서 비율 상승
- **환각 패턴 2건**: "횟집 추천" 질문에 도구 호출 없이 "취급 매장 없음" 단정
- **empty_result 1건**: 매장명 부분검색이 중간에 0건 → 재시도로 최종 성공
  — 즉 false positive. 플래그 로직에 "최종 성공 시 제외" 정교화 필요 판정

리포트가 곧바로 개선 항목이 됐다: ① 시스템 프롬프트에 "도구 미호출 상태에서
존재 여부 단정 금지" 규칙 추가, ② 매장 검색에 유사도(pg_trgm) 폴백,
③ 플래그 false positive 보정.

## 배운 것

- LLM 품질 관리의 시작은 평가 모델이 아니라 **구조화 로그**다. 도구 호출 여부와
  결과 행 수만 기록해도 환각의 상당수를 기계적으로 걸러낼 수 있다.
- "없다"고 단정하는 응답은 반드시 도구 호출 근거와 짝지어 검증하라 —
  가장 그럴듯하고 가장 위험한 답변 유형이다.
- 플래그는 처방 기준으로 분류해야 한다. 프롬프트 문제 / 데이터 문제 / 코드 문제는
  담당도 해법도 다르다.
- 표본이 작을 땐 수치보다 **개별 사례 리뷰**가 가치 있다. 주간 5건이면 전수 검토가 된다.
