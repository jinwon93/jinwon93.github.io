수백 개 매장의 메뉴 47,526건을 주류/음료/음식/세트/사이드로 자동 분류해야 했다.
LLM(Claude Haiku)에게 시키기로 했는데, 문제가 둘 있었다 — **비용**과
**서버리스 60초 타임아웃**. 둘 다 설계로 풀었다.

## 비용: 프롬프트에서 1/20 절감

처음 설계는 전체 코드 트리(전 카테고리)를 프롬프트에 넣고 분류를 맡기는
방식이었다. 견적을 내보니 약 $30. 그런데 생각해보면 "메뉴 종류" 분류에
재료·조리법 코드까지 보여줄 이유가 없다.

- 카테고리별 재태깅 모드를 만들어 **해당 카테고리 코드만 프롬프트에 전달**
  → 입력 토큰 1/20, 비용 $30 → **$2~4**
- 고유 메뉴명 15개씩 배치, 메뉴당 최대 2개 코드만 반환하도록 제약
- 식기·포장용기 같은 비매출 항목은 LLM에 보내기 전에 SKIP_LIST로 걸러냄 —
  모델 호출 자체가 비용이다

## 타임아웃: fire-and-forget tick 체인

47,526건을 한 요청에서 처리하면 서버리스 60초 제한에 걸린다. 그래서 러너를
**자기 자신을 다시 호출하는 tick 체인**으로 만들었다.

```
start → tick(5배치 처리) → 자기 자신 재호출 → tick → ... → 완료
```

- 실행 상태 테이블에 **락(`is_running`) + resume 지점(`last_processed_page`)** 기록
- 중복 실행 방지, 중단돼도 다음 start가 이어서 재개
- 관리 화면에서 start/stop + 3초 폴링으로 진척률 표시, 진척률은 RPC로 집계

## 결과와 함정들

태깅 완료율 **97.8%** (고유 메뉴 15,479/15,829). 표본 검증에서 사케(준마이긴죠),
전통주(산사춘), 세트(콤보) 모두 정확했다. 잔여 2.2%는 숫자코드·식기·테스트
항목이라 통계에 영향이 없는 것으로 판정하고 종료했다.

도중에 밟은 함정 둘: API 게이트웨이(PostgREST)의 **기본 1000행 limit**에 코드
테이블(1,086행)이 잘려 조용히 오분류가 났고 — 페이지네이션 헬퍼로 해결.
중첩 조인 문법 불일치로 500이 났는데 에러가 삼켜져 있어서, 전체 try/catch에
실제 메시지·스택을 노출시키고서야 원인을 잡았다.

마지막으로 **단일성 가드** — 이미 "메뉴 종류" 태그가 있는 메뉴에 다른 경로
(수동·키워드 룰)로 또 태깅되는 것을 차단하고, 한 배치에서 한 메뉴에 종류 2개가
반환되면 1개만 채택하게 했다. 분류 축마다 하나라는 불변량은 코드가 지켜야 한다.

## 배운 것

- LLM 배치의 비용은 모델이 아니라 **프롬프트 설계**에서 절감된다. 필요한 문맥만 줘라.
- 서버리스에서 긴 작업은 "한 방"이 아니라 **락 + resume 지점을 가진 체인**으로.
- 게이트웨이의 기본 limit은 에러 없이 데이터를 자른다. 행 수를 검증하라.
- 불변량(메뉴당 종류 1개)은 어느 경로로 태깅되든 지켜지도록 가드를 공용화하라.
