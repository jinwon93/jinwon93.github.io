스프레드시트로 굴러가던 장애접수(A/S) 업무를 사내 통계 시스템 안에 올려달라는
요구가 왔다. 3년치 4,842건이 쌓인 구글 시트가 원장이었고, 목록·검색·접수·통계까지
붙여야 했다. 며칠 만에 1차가 돌기 시작하자 주변에서 물었다 — "어떻게 이렇게 빨리
만들었냐." 답은 반전이었다. **빨리 만든 게 아니라, 오래 걸리는 부분이 이미
만들어져 있었다.** 그 "이미 있던 것"의 정체를 정리한 기록이다.

## 오래 걸리는 부분은 화면이 아니다

사내 시스템을 새로 만들 때 실제로 시간을 잡아먹는 건 업무 화면이 아니라 그
아래 기반이다. 로그인과 권한, 공통 기준 데이터, 보안 규칙, DB 설계 표준 —
새 업무 하나 붙일 때마다 이걸 매번 다시 고민하면 며칠이 몇 주가 된다.
이번엔 그게 전부 통계 플랫폼을 만들며 갖춰져 있었다.

- **인증·권한**: 역할 기본 × 개인 예외의 2축 모델(별도 글)이 이미 돌고 있었다.
  새 업무용 역할 하나를 목록에 추가하고, 페이지×역할 매핑 테이블에 행 두 개를
  넣은 게 권한 작업의 전부였다. 로그인·세션·미들웨어 가드·관리 UI는 손대지 않았다.
- **공통 기준 데이터**: 회사 전체가 공유하는 매장 원장이 이미 있었다. 매장마다
  고유 번호가 있고, 매출·메뉴·예약이 전부 이 키로 조인되고 있었다. 새 도메인은
  매장 테이블을 새로 만드는 대신 이 원장에 **연결만** 했다.

즉 새로 만든 건 장애접수 테이블 두 개, 집계 함수 몇 개, 화면뿐이었다.

## 규약이 있으면 새 테이블은 고민이 없다

기반보다 더 조용히 시간을 아껴준 건 "새 테이블을 만들 때의 팀 규약"이었다.
세 가지가 이미 문서가 아니라 몸에 배어 있었다.

**① RLS 전면 활성 — 새 테이블도 예외 없이.** 모든 테이블에 Row Level Security를
켜는 게 규약이다. 새 도메인 테이블은 정책을 **하나도 만들지 않았다.** 이건
"일반 클라이언트 키로는 아무 행도 못 읽고, 오직 서버만 접근 가능"이라는 뜻이다.
브라우저가 DB에 직접 붙는 경로를 원천 차단하고 반드시 앱 서버를 거치게 만드는
잠금이다. 권한 판정은 앱(위의 2축 모델)이 하고, DB는 뒷문을 닫아둔다.

**② 함수 search_path 고정 — 보안 기본값.** Postgres 함수는 실행 시 테이블을
어느 스키마에서 찾을지 탐색 경로를 따라간다. 이걸 고정하지 않으면 같은 이름의
가짜 테이블을 다른 스키마에 심어 함수를 속이는 공격이 성립한다. 그래서 모든
함수에 경로를 못 박는 게 규약이다. 새로 짠 검색·집계·유사사례 함수도 전부 이
한 줄을 달고 나왔다 — "이거 보안상 괜찮나?"를 다시 고민할 필요가 없었다.

**③ 인덱스 플레이북 — 새 테이블용 체크리스트.** 날짜 조회가 주력이면 날짜
내림차순, 상태·분류 같은 필터 컬럼에 인덱스, 텍스트 검색은 trigram GIN, 조인
키에는 커버링 인덱스. 이 목록대로 처음부터 걸었다.

```sql
-- 새 도메인 테이블 = 규약대로 찍어낸 결과 (식별자는 일반화)
alter table ops_ticket enable row level security;   -- 정책 없음 = 서버 전용

create function search_tickets(q text) returns setof ops_ticket
  language sql stable
  set search_path = public, pg_temp             -- ② 보안 규약
as $$ ... $$;

create index on ops_ticket (occurred_at desc);      -- ③ 날짜 주력
create index on ops_ticket (status);                -- ③ 필터 컬럼
create index on ops_ticket using gin (search_text gin_trgm_ops);  -- ③ 텍스트 검색
```

덕분에 4,800여 건 규모의 검색·통계가 전부 수 ms에 나오고, 나중에 수십만 건이
돼도 구조를 바꿀 필요가 없다. 규모도 부담이 아니었다 — 새 도메인 테이블 전체가
14MB로, 11GB짜리 전체 DB의 0.12%에 불과했다. 기존 통계 부하에 사실상 영향이 없다.

## 진짜 공수는 데이터 정제였다

그래서 실제로 시간이 어디로 갔느냐. 코드가 아니라 **스프레드시트 정제**였다.
공수의 8할이 여기 있었다.

- **컬럼 밀림 보정**: 사람이 손으로 채운 시트라 특정 행부터 열이 한 칸씩 밀려
  있었다. 매장명 열에 접수자 이름이 들어가는 식. 파싱 규칙을 실측으로 맞췄다.
- **표기 통일**: 상태·분류 값이 자유 입력이라 같은 뜻이 여러 표기로 흩어져 있었다.
- **빈 행 제거**: 헤더·주석·빈 줄 등 900여 건을 걷어내고 4,842건이 남았다.
- **매장명 정규화 매칭**: 시트엔 매장이 그냥 텍스트였다. 정규화 후 원장의 고유
  번호에 자동 매칭시켜 **92%**가 붙었고, 나머지는 별도 연결 화면에서 수동 처리했다.

이 매칭이 붙는 순간 장애 건수가 매출·예약과 **같은 좌표계**에 올라간다. "이 매장의
매출"과 "이 매장의 장애 건수"를 한 키로 묶을 수 있게 되는 것이다.

## 무엇이 얼마나 걸렸나

| 구분 | 한 일 | 체감 공수 |
|---|---|---|
| 인증·권한 | 역할 1개 + 매핑 행 2개 추가 | 거의 0 |
| 매장 기준 데이터 | 기존 원장에 연결만 | 낮음 |
| 보안·인덱스 | 규약대로 테이블·함수 생성 | 낮음 |
| 화면·집계 함수 | 신규 개발 | 중간 |
| **데이터 정제** | 밀림 보정·표기 통일·매칭 | **8할** |

## 배운 것

- "빨리 만들었다"의 정체는 대개 재사용이다. 새 도메인이 빨랐던 건 인증·기준
  데이터·보안·인덱스가 이미 플랫폼이었기 때문이지, 이번에 잘 짜서가 아니다.
- 규약의 가치는 "고민을 없애는 것"이다. RLS·search_path·인덱스를 매번 결정하지
  않고 찍어낼 수 있으면, 새 테이블 하나의 리드타임이 시간 단위에서 분 단위로 준다.
- 기반이 갖춰지면 개발 속도의 병목은 코드가 아니라 **데이터 품질**로 옮겨간다.
  결국 시트를 얼마나 깨끗이 원장 키에 붙이느냐가 전체 속도를 결정했다.
- 새 업무를 얹기 전에 물어라 — "이건 새로 만들 일인가, 기존 규약에 태울 일인가."
  대부분은 후자다.
