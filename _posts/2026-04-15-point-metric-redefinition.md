포인트(적립/사용/취소) 통계 페이지를 만들려고 기존 집계 쿼리를 열었다가
이상한 걸 발견했다. 원본 테이블의 적립은 6,027건인데 기존 통계는 3,486건 —
**거의 절반이 증발**해 있었다. 버그가 아니라, 지표 정의 자체가 잘못돼 있었다.

## 원인: "적립 건수"가 아니라 "주문에 매칭된 적립"을 세고 있었다

기존 쿼리는 적립 행을 주문 로그와 **금액 역산 + ±150초 시각 fuzzy 매칭**으로
연결해, 매칭에 성공한 것만 세고 있었다. 문제는 적립이 주문과 무관하게도
발생한다는 것 — 신규 가입 지급, 매장 수기 지급, 시스템 이관. 이들은 매칭될
주문이 없으니 전부 누락됐다. 게다가 취소(act=3)는 아예 집계 대상에 없었다.

원본 담당 개발자에게 코드 의미를 확정받았다: `act 1=적립 / 2=사용 / 3=취소`,
그리고 "적립은 주문과 별개"라는 답. 지표를 다시 정의했다 —
**"적립된 주문 수"가 아니라 "적립 발생 건수"**. 매장 점주가 실제로 궁금한 것도 후자다.

## 재구축: 미러 → 일집계 → RPC

- **미러 테이블**: 원본을 그대로 복제하되, 전화번호 등 **PII 컬럼은 SELECT
  단계에서부터 제외** — 애초에 분석 DB로 넘어오지 않게
- **영업일 귀속**: 06:00 이전 거래는 전날 영업일로 — STORED generated column으로
  적립/사용/취소 전부에 **동일 규칙** 적용 (기존엔 지표마다 컷오프가 달랐다)
- **일집계 테이블**: `(매장, 날짜)` PK에 적립/사용/취소 × 건수/금액 6지표,
  `COUNT(*) FILTER (WHERE act=N)` 분기, 멱등 DELETE+INSERT 빌더
- **증분 sync**: 주문과 달리 생성 시각밖에 없어 워터마크 − 10분 오버랩 재스캔
  방식 — 원본 insert 지연·역순 도착을 흡수

## 검증: 숫자가 달라진 이유를 전부 설명할 것

| 지표 | 기존 | 신규 | 차이의 정체 |
|---|---|---|---|
| 적립 | 3,486 | **6,027** | fuzzy 누락분(가입·수기·이관 적립) 복원 — 신규가 정확값 |
| 사용 | 89 | 101 | 기존 쿼리의 날짜 경계 누락 + 06시 컷오프 차이 |
| 취소 | (미집계) | 38 | 신규 확보 지표 |

전체 sync는 118,362행 / **15.3초** — 기존 fuzzy 쿼리는 실행에만 5분이 걸렸다.
구조를 고치니 성능은 따라왔다.

## 배운 것

- 성능보다 먼저 물을 것: **"이 숫자는 무엇을 세고 있는가."** 빠르게 틀린 답을
  주는 시스템이 제일 위험하다.
- 원본 코드값의 의미는 추측하지 말고 소유자에게 확정받아라. `act=3`처럼
  아예 빠진 축이 있을 수 있다.
- PII는 마스킹이 아니라 **수집 단계 차단**이 가장 싸고 확실하다.
- 지표를 바꿀 땐 신구 수치의 차이를 **전부 설명**하고 배포하라. "새 숫자가
  더 크니 좋다"는 검증이 아니다.
