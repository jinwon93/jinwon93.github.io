재로그인 시 인증 API가 9.1초 걸린다는 제보가 왔다. 첫 반응은 당연히 "DB가 느린가?"였다.
결론부터: DB는 무죄였고, 진짜 범인을 찾는 과정에서 더 큰 물고기 세 마리를 잡았다.

## 판정: 감쇠 패턴을 읽어라

같은 API를 연속 호출하며 응답 시간을 쟀다. **9.1초 → 6.2초 → 1.0초 → 78ms.**
이 감쇠 곡선이 진단의 전부였다. DB가 느리면 매번 느려야 한다. 첫 호출만 느리고
급격히 정상화되는 패턴은 **서버리스 콜드 스타트**의 지문이다. 재로그인을 다시
실측하니 웜 상태에서 112ms — DB 무죄 확정.

교훈: "느리다"는 제보를 받으면 한 번 재지 말고 **연속으로 재라.** 패턴이 원인을 말해준다.

## 덤으로 잡은 것 1: bpchar vs text 함정

진단 중 pg_stat_statements에서 무거운 크론 배치들이 눈에 띄어 내친김에 털었다.
가장 극적이었던 건 타입 불일치 한 줄.

컬럼은 `char(8)`(bpchar)인데 함수 변수가 `text`였다. 이 미묘한 차이 때문에
인덱스 시크가 불가능해져 배치가 6.9초씩 걸리고 있었다. 캐스팅 한 줄로 해결:

```sql
-- before: WHERE order_date >= v_cutoff          (text — 인덱스 못 탐)
-- after:  WHERE order_date >= v_cutoff::bpchar  (인덱스 시크, 6.9s → 158ms)
```

교묘한 점은 hypopg 가상 인덱스 테스트에선 **리터럴로 시험해서 통과**했다는 것.
실제 함수는 변수를 쓰니 실행 경로가 달랐다. 도구는 실제 실행 형태로 검증해야 한다.

## 덤 2: 인덱스가 답이 아닐 때

또 다른 배치는 hypopg로 인덱스를 제안해도 플래너가 거부했다. 원인은 선택도 —
조회 윈도우(14일)가 롤링 보존(50일) 전체의 27%라, 플래너 입장에선 풀스캔이 합리적이었다.
해법은 인덱스가 아니라 **쿼리 재구성**: 수학적으로 동치인 날짜 조건 푸시다운 +
조인 전략 유도로 hash anti join을 태웠다.

## 덤 3: 증분화는 "동일함의 증명"과 함께

일별 배치 두 개를 풀스캔에서 증분 방식으로 전환하면서, 전환 전후 결과를
`EXCEPT` 양방향 대조로 검증했다 — **차이 0행**(순위까지 91,496행 일치)을 확인하고
배포. 연관 페어 집계의 동점(tie) 항목이 실행마다 순서가 바뀌는 문제는 결정적
tiebreaker를 추가해 재현성을 확보했다.

| 배치 | 전 | 후 |
|---|---|---|
| 헬스 메트릭 (매시간) | 45.6s | 8.9s (일 ~75GB I/O 제거) |
| 추천 연관도 빌드 (일별) | 86s | 23s |
| MV 갱신 3종 | 162s | 30s |
| 인증 API (웜) | 9.1s (콜드) | 112ms |

## 배운 것

- 응답 시간의 **감쇠 패턴**은 콜드 스타트/캐시/경합을 구분해주는 공짜 진단 도구다.
- 타입 불일치(bpchar↔text)는 조용히 인덱스를 무력화한다. 실행 계획에서 캐스트를 의심하라.
- 인덱스 제안 도구가 거부당하면 선택도를 보라. 답이 쿼리 재구성일 때가 있다.
- 배치 로직을 바꿀 땐 "빨라졌다"만으론 부족하다. **결과가 같음을 증명**하고 배포하라.
