메뉴 별칭 시스템(표기가 다른 같은 메뉴를 하나로 병합)을 운영에 넘기기 전,
운영자가 실수할 수 있는 경로를 하나씩 막은 작업의 기록이다. 화려한 기능보다
이런 보강이 시스템의 수명을 결정한다고 믿는다.

## 문제 1: 별칭 체인(transitive alias)

별칭 구조는 단순하다. `alias_norm → canonical_norm` 매핑이고, 통계는 조회 시점에
`COALESCE(alias.canonical_norm, norm_name)`으로 **1단계만** 해석한다. 여기에 함정이 있다.

A→B로 묶었는데 누군가 B→C를 등록하면? A는 B로, B는 C로 가는데 해석은 1단계라
**A와 C가 다른 그룹으로 갈라진다.** 통계가 조용히 틀어지는 최악의 유형이다.

DB 제약(`UNIQUE(alias_norm)`, self-loop CHECK)은 직접 루프만 막지 체인은 못 막는다.
CHECK는 다른 행을 볼 수 없기 때문이다. 그래서 **API 계층에 사전검사 2종**을 넣었다.

- 역방향: 지정한 canonical이 이미 다른 것의 alias인가 → 409 + "대신 ○○를 대표로 사용하세요"
- 정방향: 등록하려는 alias가 이미 다른 별칭들의 canonical인가 (하위 별칭 끊김 방지)

부수 디테일: 메뉴명엔 괄호·쉼표가 흔해서 쿼리 빌더의 `.or()` 문자열 보간이 위험했다.
명시적 2쿼리 + 병렬 실행으로 우회했다.

## 문제 2: 조용히 삼켜진 에러

병합 미리보기가 매출이 분명히 있는 메뉴(판매 2.8만 건)인데 "데이터 없음"으로 떴다.
원인은 `fetch(...).catch(() => {})` — **실패를 조용히 삼키고 빈 상태로 렌더**하고
있었다. 로딩/에러/빈값 3상태를 분리하고 실패는 실패라고 표시하게 고쳤다.
"데이터 없음"과 "불러오기 실패"는 완전히 다른 정보다.

## 문제 3: 보이지 않는 현재 상태

병합 화면에서 그 대표에 **이미 무엇이 묶여 있는지** 안 보였다. 중복 등록과
오등록의 온상이다. 현재 구성원을 칩으로 노출하고 X로 즉시 해제, 등록된 별칭은
대표별 폴더 카드로 그룹화했다. 탭 이름도 상태 중심으로 바꿨다 —
"병합 추천"(아직 할 일) / "자동 병합됨"(이미 끝난 것).

## 배운 것

- **무결성 규칙에는 층이 있다.** 행 단위 규칙은 DB 제약으로, 행 사이를 보는 규칙은
  API 가드로. 후속으로 DB 트리거 원천 차단도 계획에 넣었다 — 겹겹이 막는 게 맞다.
- `.catch(() => {})`는 버그를 숨기는 코드다. 에러 상태는 반드시 표면화하자.
- 파괴적 작업 UI는 "실행 전 현재 상태"를 보여주는 것만으로 사고가 크게 준다.
